## 概要
- モックアップ（Mockup）とは、Webアプリの画面デザインや構成を視覚的に表現した「静的な見本」のこと
- ワイヤーフレーム（骨格）よりも高い忠実度で、実際の色・フォント・画像配置などを含む
- コーディング前に「完成イメージの合意形成」を目的として作成される
- プロトタイプ（動く試作品）とは異なり、基本的にインタラクションは持たない

## 結論
- モックアップはチーム内・顧客との認識齟齬を防ぎ、手戻りコストを削減する最も費用対効果の高い手段
- 「見た目の確認」だけでなく、UX上の問題を早期に発見する役割も担う
- 開発着手前に必ずステークホルダーの承認を得ることが重要

## 要点
- **忠実度（Fidelity）の段階**：ローファイ（手書き・簡易）→ ミッドファイ（ワイヤーフレーム）→ ハイファイ（モックアップ）→ プロトタイプ
- **作成タイミング**：要件定義完了後、実装前。デザインスプリントの一環として行うことが多い
- **主な作成手法**：HTML/CSS静的ページ（古くからある手法）、Figma・Adobe XD・Sketch（現代的な設計ツール）など。手法はチームや時代によって異なる
- **含む要素**：レイアウト・配色・タイポグラフィ・アイコン・画像の仮配置・ナビゲーション構造
- **含まない要素**：実際の動作・API連携・データの流れ（これらはプロトタイプ・実装で扱う）
- **レビュープロセス**：デザイナー → PO/PM → ステークホルダー → 開発者の順でフィードバックを収集するのが一般的

## 詳細

### モックアップの位置づけ（開発フローにおける）
```
要件定義 → IA設計 → ワイヤーフレーム → モックアップ → プロトタイプ → 実装 → テスト
```
モックアップは「見た目の最終確認」の段階に位置し、この段階での変更コストは実装後の数分の一に抑えられる。

### ローファイ・ハイファイの使い分け
| 種類 | 特徴 | 用途 |
|------|------|------|
| ローファイ | 手書き・ボックスのみ | アイデア出し・初期議論 |
| ミッドファイ | グレースケール・構造重視 | 情報設計の確認 |
| ハイファイ | 実際の色・画像・フォント | デザイン承認・開発仕様書 |

### モックアップ作成手法の変遷（時系列）

| 時代 | 主な手法 | 特徴・背景 |
|------|----------|------------|
| 〜1990年代 | 紙・手書きスケッチ、PowerPoint | デジタルツールが乏しく、紙が主流 |
| 2000年代前半 | Dreamweaver、Visio、HTML/CSS静的ページ | Web制作者がそのままHTMLでモックを作る慣行が普及 |
| 2000年代後半〜2010年代前半 | Fireworks、Balsamiq、Axure RP、HTML/CSSモック | 専用ツールが登場するが、HTMLモックも並行して利用 |
| 2011年〜 | Bootstrapの登場でHTMLモックの作成コストが大幅低下 | フロントエンド主導のモック作成が加速 |
| 2010年代後半 | Sketch（Mac専用）が台頭 | デザイン専用ツールへの移行が進む |
| 2016年〜現在 | Figmaが主流化 | ブラウザベース・リアルタイム共同編集・非エンジニアも扱いやすい |

> **補足**：HTML/CSSによる静的ページ作成は、Figma登場以前から「HTMLモック」として業界で広く使われてきた正当な手法。現代でもフロントエンドエンジニア主導のプロジェクトでは有効な選択肢。

### Figmaを使った現代的なワークフロー（参考）
1. コンポーネントライブラリ（デザインシステム）の準備
2. 各画面のフレームを作成
3. オートレイアウトでレスポンシブ対応を考慮
4. スタイル（カラー・テキスト）を変数化してメンテナンス性を確保
5. コメント機能でフィードバックを集約
6. 開発者モードで寸法・CSSの自動出力

### よくある失敗パターン
- モックアップを「絵」として扱い、インタラクションや状態変化（エラー表示・ローディング等）を省く
- ステークホルダーの承認を得ずに実装に進み、手戻りが発生する
- コンポーネントを使い回さず、修正時に全画面を個別更新する羽目になる
- スマートフォン・PCそれぞれのブレークポイントを想定しないまま作成する

## 実務でどう使うか
- **PBL（Project-Based Learning）文脈**：チームで要件を固めた後、Figmaで画面設計を共有し、認識を揃えてから実装に入る
- **レビュー依頼時**：「このモックアップで実装して問題ないか」を明示的にチームに確認してから次工程へ進む
- **仕様書としての活用**：開発者はモックアップからCSS値・余白・フォントサイズを直接参照できるため、口頭指示を減らせる
- **変更管理**：Figmaのバージョン履歴を活用し、どの時点のデザインかをトレースできるようにする
- **アクセシビリティ考慮**：コントラスト比・フォントサイズ・タッチターゲットサイズをモックアップ段階で確認する
